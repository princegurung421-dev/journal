Week 6

Flask Backend Integration & Frontend-Backend Connection

December 2025 Flask & Backend

This week focused on integrating Flask as a backend framework for the Learning Journal PWA. The primary goal was to establish a connection between the frontend (browser interface) and backend (server) to transform the PWA from a static, local document into a dynamic, persistent web application. I implemented RESTful API endpoints using Flask and connected them to the frontend using JavaScript's Fetch API.

Why is the frontend–backend connection important?

The connection between the frontend (the browser interface) and the backend (the server) is essential because it moves the PWA from being a static, local document to a true, dynamic application. Before this lab, my journal entries were only saved to my browser's localStorage. That data was locked to that specific computer and browser; it wasn't persistent or shareable. The backend, built with Flask, provides the persistence by saving the data to a file (reflections.json) on the server. This makes the application useful because the data is now secure, available from any device, and can be managed centrally, which is the whole point of a web application.

Which HTTP methods did you use in Flask, and why?

I implemented three HTTP methods in my Flask backend: GET, POST, and DELETE.

  • GET was used for retrieving data. When the PWA loads, the JavaScript makes a GET request to /api/reflections to load and display all saved journal entries.
  • POST was used for creating new data. When I submit the journal form, the JavaScript sends a POST request with the new entry's data to /api/add_reflection to save it to the reflections.json file.
  • DELETE was used for my extra feature. It's the correct method for telling the server to remove a specific entry from the data file.

What is the difference between using Flask to store and load JSON data and reading JSON directly in the browser?

The main difference is about data management and persistence.

When reading JSON directly in the browser (or using localStorage), the data is client-side. The browser handles everything. This is simple, but the data isn't persistent across devices and can't be shared. Any changes are local.

By using Flask, the data is server-side. The browser only sends a request to the server, and Flask manages the data file (reflections.json). This ensures the data is persistent, safe, and available to anyone accessing the application. It shifts the responsibility for data integrity and security away from the user's device and onto the controlled server environment.

Did you face any difficulties when running your project on PythonAnywhere? How did you handle them?

I definitely hit a few snags, mostly related to file paths.

My biggest issue was the paths for my static assets (CSS/JS). When I first loaded the page, it was just plain HTML because the server couldn't find my stylesheet and script files. I initially used relative paths like href="css/style.css", but Flask requires template rendering. I solved this by going into my HTML file and replacing all the static file links with the correct template syntax: {{ url_for('static', filename='css/style.css') }}. After changing all the links and remembering to reload the web app in the PythonAnywhere 'Web' tab, the styles finally loaded.

What extra feature did you build into your PWA with Flask, and why did you add it?

For the extra feature, I added the ability to Delete an existing reflection.

To implement this, I had to update two things: first, I added a "Delete" button next to every reflection in the frontend. Second, I created a new @app.route() in flask_app.py that specifically handled the DELETE request. The JavaScript now sends a fetch request using the DELETE method along with a unique identifier for the reflection. This backend route removes that entry from the list and overwrites the reflections.json file.

I chose the delete feature because a journal is a personal tool that requires maintenance. If I accidentally make a duplicate entry or write something that's no longer relevant, I need a way to clean up the data. It was essential for moving my PWA toward a fully functional and practical application.

Key Learnings

  • Flask's route decorators make API creation straightforward and intuitive
  • RESTful design principles improve API usability and follow web standards
  • Fetch API enables seamless frontend-backend communication
  • Template rendering with url_for() is essential for Flask static file serving
  • Server-side data management provides persistence and security advantages
  • PythonAnywhere deployment requires proper file structure and WSGI configuration
  • HTTP methods (GET, POST, DELETE) have specific semantic meanings in REST APIs

Week 7

PWA Enhancement & Offline Support

December 2025 PWA & Service Workers

This week focused on transforming the Flask-powered Learning Journal into a full Progressive Web App (PWA) with offline access, installability, and improved user experience. I implemented a service worker, added a manifest file, and integrated caching strategies to ensure that users can still access previous reflections even when they are offline.

Why is it useful to enhance your Flask app with PWA features?

Adding PWA features makes the Flask app feel like a native application. The user can install it on their home screen, open it in full-screen mode, and continue using the app even without an internet connection. This is especially valuable for a journal application, since reflections should always be accessible. PWAs improve reliability, speed, and user engagement without needing a full mobile app.

What did you use to support offline access and dynamic data?

I implemented a service worker with two caching strategies:

  • Cache First for static assets such as CSS, JS, fonts, and images
  • Network First for the dynamic reflections API so users see the latest data but still get offline fallback

I also created a manifest.json, which provides icons, app name, short name, colors, and display mode. Together, these PWA technologies allow the app to work offline, load faster, and behave like a native mobile app.

What extra feature did you add, and why?

For this week’s extra feature, I added Offline Reflection Creation.

When the user is offline and tries to submit a reflection, the entry is saved to localStorage with a “pending” status. When the device reconnects, a small background script automatically syncs the pending reflections to the Flask backend using the POST API.

I added this because it prevents data loss and allows users to keep journaling without worrying about their network status. It makes the PWA much more durable and user-friendly.

Did you face any challenges deploying your PWA, and how did you solve them?

Yes, deployment brought two main issues:

  • Service Worker Not Updating: PythonAnywhere aggressively cached old files. I solved this by adding a version number to the cache name and using self.skipWaiting() to force the new service worker to activate.
  • HTTPS Requirement: PWAs require HTTPS for service workers. Luckily, PythonAnywhere automatically provides HTTPS, but I had to update all absolute URLs to make sure everything loaded securely.

After clearing the old cache in the browser and reloading the app, the PWA worked as expected.
